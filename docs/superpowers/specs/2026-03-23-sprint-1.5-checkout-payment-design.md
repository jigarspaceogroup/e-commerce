# Sprint 1.5 ‚Äî Checkout & Stripe Payment Design Spec

**Date**: 2026-03-23
**Branch**: `feature/sprint-1.5-checkout-payment`
**Tasks**: TASK-098 through TASK-111

---

## 1. Overview

Sprint 1.5 adds the complete checkout flow: order creation from cart, Stripe card payment, webhook-driven status updates, multi-step checkout UI, guest checkout, and order confirmation. This builds on Sprint 1.4's cart, coupon, profile, and address book features.

**Key constraints**:
- Phase 1 payment method: credit/debit card via Stripe only
- Currency: SAR (amounts sent to Stripe in halalas = amount √ó 100)
- Guest checkout supported (no account required)
- Order number format: `ORD-YYYYMMDD-XXXXX` (sequential per day)
- 15% VAT, free shipping at 500+ SAR, otherwise 30 SAR flat

## 2. Database (TASK-098, TASK-099)

### 2.1 Schema Changes Required

The Prisma schema already defines `Order`, `OrderItem`, `Payment`, `PaymentEvent`, and `Refund` models. The following changes are needed before Sprint 1.5 implementation:

**Add `OrderStatusHistory` model** (missing entirely):
```prisma
model OrderStatusHistory {
  id         String       @id @default(uuid()) @db.Uuid
  orderId    String       @map("order_id") @db.Uuid
  fromStatus OrderStatus? @map("from_status")
  toStatus   OrderStatus  @map("to_status")
  changedBy  String?      @map("changed_by") @db.Uuid
  notes      String?
  createdAt  DateTime     @default(now()) @map("created_at")

  order Order @relation(fields: [orderId], references: [id], onDelete: Cascade)

  @@index([orderId])
  @@map("order_status_history")
}
```
Add `statusHistory OrderStatusHistory[]` relation on the `Order` model.

**Add `gatewayEventId` to `PaymentEvent`** (needed for webhook idempotency):
```prisma
gatewayEventId String? @unique @map("gateway_event_id")
```

**Change `OrderItem.productTitleSnapshot` from `String` to `Json`** (to store `{ en, ar }` bilingual titles):
```prisma
productTitleSnapshot Json @map("product_title_snapshot")
```

**Make `Order.billingAddress` optional** (Phase 1 does not collect billing address separately ‚Äî defaults to shipping address):
```prisma
billingAddress Json? @map("billing_address")
```

**Add `idempotencyKey` to `Order`** (for checkout idempotency, separate from Payment's idempotency key):
```prisma
idempotencyKey String? @unique @map("idempotency_key")
```

**Guest phone**: No dedicated `guestPhone` field. Guest phone is captured in the shipping address JSON snapshot (`phone` field within the address object). This is sufficient since every address includes a phone number per the existing address schema.

### 2.2 Migration

Run `prisma migrate dev --name sprint-1.5-checkout-payment` after applying all schema changes. Changes are additive (new table, new columns, one type change from String to Json on productTitleSnapshot).

## 3. Backend Services

### 3.1 Checkout Service (`services/checkout.ts`) ‚Äî TASK-100

**`createOrder(params)`**

Parameters:
```typescript
interface CreateOrderParams {
  cartId: string;
  userId?: string;
  guestEmail?: string;
  shippingAddressId?: string;   // for authenticated users with saved address
  shippingAddress?: GuestAddress; // for guests or new address
  idempotencyKey: string;
}
```

Flow:
1. **Idempotency check**: Query `Order` by `idempotencyKey` field. If found, return existing order (200, not error).
2. **Validate cart**: Fetch cart with items. Throw if empty.
3. **Stock & price validation**: For each cart item, fetch current variant. Compare `effectivePrice` (priceOverride ?? basePrice) against `unitPriceAtAddition`. Compare requested quantity against `stockQuantity`. If any mismatch, return error with details per item (which prices changed, which items are out of stock).
4. **Resolve shipping address**: If `shippingAddressId`, fetch from address service and verify ownership. If `shippingAddress` (guest), validate via Zod. Snapshot the address as JSON.
5. **Billing address**: Set `billingAddress` to `null` in Phase 1. The shipping address serves as the billing address. The field is nullable to support future billing address collection.
6. **Calculate totals**: Reuse cart's calculated values (subtotal, discountAmount, taxAmount, shippingEstimate as shippingFee, grandTotal). These were already computed by `recalculateCart`.
7. **Generate order number**: Use Redis `INCR` on key `order-seq:{YYYYMMDD}`. On first use each day, set expiry via `EXPIREAT` to next midnight UTC+3 (Saudi time) to avoid day-boundary edge cases. Format: `ORD-{YYYYMMDD}-{seq.toString().padStart(5, '0')}`.
8. **Create in transaction** (`prisma.$transaction`):
   - Create `Order` with all fields, address snapshot, coupon code from cart, `idempotencyKey`
   - Create `OrderItem` for each cart item with product snapshots: `productTitleSnapshot` as `{ en: titleEn, ar: titleAr }` (Json), `variantAttributesSnapshot` (Json), `skuSnapshot`, `unitPrice`, `quantity`, `lineTotal`
   - Decrement `stockQuantity` on each `ProductVariant` using optimistic locking (`UPDATE ... WHERE stockQuantity >= requestedQty`). If any row returns 0 affected, throw `STOCK_INSUFFICIENT` and roll back.
   - Create initial `OrderStatusHistory` entry (fromStatus: null, toStatus: `pending_payment`)
   - Delete all `CartItem` records for this cart
   - Reset cart totals to zero and clear couponCode
9. **Return**: Order with items and order number.

**Error scenarios**:
- Cart empty ‚Üí 400 `CART_EMPTY`
- Stock insufficient ‚Üí 409 `STOCK_INSUFFICIENT` with item details
- Price changed ‚Üí 409 `PRICE_CHANGED` with old/new prices per item
- Duplicate idempotency key ‚Üí return original order (200, not error)

### 3.3 Stripe Payment Service (`services/stripe-payment.ts`) ‚Äî TASK-102

```typescript
import Stripe from 'stripe';
const stripe = new Stripe(process.env.STRIPE_SECRET_KEY!);
```

**Methods**:

**`createPaymentIntent(orderId, amount, currency, idempotencyKey)`**:
1. Create Stripe `PaymentIntent` with:
   - `amount`: in halalas (SAR √ó 100, e.g., 150.50 SAR = 15050)
   - `currency`: 'sar'
   - `metadata`: `{ orderId, idempotencyKey }`
   - `idempotencyKey` on the Stripe API call
   - `capture_method`: 'automatic' (auto-capture, no separate capture step needed in Phase 1)
2. Create `Payment` record: status `pending`, `gatewayTransactionId` = PaymentIntent ID, `idempotencyKey`
3. Log `PaymentEvent`: eventType `initiated`, eventData = Stripe response
4. Return `{ clientSecret: paymentIntent.client_secret, paymentId: payment.id }`

**`getPaymentStatus(paymentId, userId?)`**: Query Payment record and return current status. Verify ownership via order's userId.

**Note on dev plan TASK-102**: The dev plan mentions `confirmPayment` and `capturePayment` methods. In Phase 1, we use `capture_method: 'automatic'`, so capture happens automatically on confirmation. `confirmPayment` happens client-side via `stripe.confirmCardPayment()`. No server-side confirm/capture methods are needed. These can be added in Phase 2 if authorization-then-capture workflow is required.

**Error mapping**: Map Stripe error codes to user-friendly messages:
- `card_declined` ‚Üí "Your card was declined. Please try another card."
- `expired_card` ‚Üí "Your card has expired."
- `insufficient_funds` ‚Üí "Insufficient funds."
- `processing_error` ‚Üí "An error occurred processing your card. Please try again."
- Default ‚Üí "Payment failed. Please try again or use a different payment method."

### 3.4 Stripe Webhook Handler (`services/stripe-payment.ts`) ‚Äî TASK-103

**`handleWebhookEvent(rawBody, signature)`**:
1. Verify signature: `stripe.webhooks.constructEvent(rawBody, signature, STRIPE_WEBHOOK_SECRET)`
2. Extract `event.type` and `event.id` (Stripe's event ID)
3. **Idempotency**: Check if `PaymentEvent` with `gatewayEventId = event.id` exists. If yes, return 200 immediately.
4. Handle events:
   - `payment_intent.succeeded`: Update Payment ‚Üí `captured`, Order ‚Üí `payment_confirmed`, log PaymentEvent with `gatewayEventId`, record OrderStatusHistory
   - `payment_intent.payment_failed`: Update Payment ‚Üí `failed`, Order stays `pending_payment`, log PaymentEvent. **Stock rollback**: restore `stockQuantity` on each variant by the order's quantities, set order status to `cancelled`, record OrderStatusHistory.
   - `charge.refunded`: Update Payment ‚Üí `refunded`/`partially_refunded`, update Order status, log PaymentEvent
   - `charge.dispute.created`: Log PaymentEvent, flag order for admin review (set `notes` field with dispute info). Full dispute handling deferred to Phase 2.
5. Unknown events: Log and acknowledge with 200

**Important**: Webhook endpoint must receive raw body (not JSON-parsed) for signature verification. Use `express.raw({ type: 'application/json' })` on this specific route.

### 3.5 Stale Order Cleanup

A scheduled job (BullMQ repeatable job) runs every 15 minutes:
- Find orders with `status = pending_payment` and `createdAt` older than 30 minutes
- For each: restore stock quantities, set order status to `cancelled`, record OrderStatusHistory with notes "Payment timeout ‚Äî stock released"
- This prevents phantom stock depletion from abandoned checkouts

**"Place Order" button flow**:

Note: The development plan (TASK-108) describes the flow as: validate ‚Üí confirm payment ‚Üí create order. However, the correct Stripe PaymentIntent flow is: create order first (to have an order to charge for), then create PaymentIntent, then confirm on client. This is the standard Stripe integration pattern.

1. Disable button immediately, show spinner
2. Call `POST /checkout` to create order (with idempotencyKey)
3. On success, call `POST /payments/initiate` with orderId ‚Üí returns `clientSecret`
4. Call `stripe.confirmCardPayment(clientSecret)` with the card element
5. **On payment success**: Poll `GET /payments/:id/status` until `captured` (max 3 attempts, 2s interval) to wait for webhook processing. If still `pending` after polling, redirect to confirmation page anyway ‚Äî the confirmation page handles `pending_payment` status gracefully with a "Payment processing..." message.
6. On payment success ‚Üí redirect to `/order-confirmation/{orderId}`
7. On payment failure ‚Üí show error message, re-enable button, offer retry or change payment method. The stale order cleanup job (Section 3.5) will cancel the order and release stock after 30 minutes.
8. On stock/price error from checkout ‚Üí show detailed error with changed items, link back to cart

**Double-click prevention**: Button disabled on first click + idempotency key on backend.

**Cart cache invalidation**: After successful order creation (step 2), invalidate the cart React Query cache: `queryClient.invalidateQueries({ queryKey: queryKeys.cart.all })`. This ensures the cart UI reflects the cleared cart if the user navigates away.

## 8. Security Considerations

- **PCI compliance**: No raw card data touches our servers. Stripe Elements handles card input. Only `clientSecret` and `paymentIntentId` cross the boundary.
- **Webhook verification**: Raw body + Stripe signature verification. Reject invalid signatures with 400.
- **Idempotency**: Order creation uses `idempotencyKey` on Order model. Payment creation uses `idempotencyKey` on Payment model and Stripe API calls.
- **Guest access control**: Guest orders accessible only via matching email + order number, or active guest session cookie.
- **Stock reservation**: Atomic decrement with optimistic locking in a transaction prevents overselling. Stale order cleanup job releases stock after 30 minutes.
- **CSRF**: Existing CORS + SameSite cookie policy protects checkout endpoints.
