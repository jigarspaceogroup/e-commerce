# Sprint 1.3 — Search & Product Discovery + Cart Foundation — Design Spec

## Overview

**Goal:** Integrate Meilisearch for full-text product search with Arabic support and typo tolerance, build the search UI with auto-suggest, implement advanced faceted filtering, and build the shopping cart foundation (API + storefront integration).

**Scope:** 17 tasks (TASK-074 through TASK-090, excluding TASK-091/092 DevOps). Covers search backend, cart backend, search frontend, cart frontend, and full test suite.

**Design System:** All new storefront UI follows the SHOP.CO design system — black-dominant palette, Satoshi/Clash Display fonts, pill-shaped radii, `<Button>`/`<Input>`/`<RatingStars>` components, `lucide-react` icons. See `docs/superpowers/specs/2026-03-19-figma-design-migration-design.md` for full reference.

## Phase 1: Search Backend

### 1.1 Meilisearch Initialization (TASK-074)

**Problem:** The Meilisearch client (`services/search.ts`) and BullMQ worker (`jobs/product-sync.ts`) exist but are never initialized on app startup. `index.ts` and `app.ts` don't call `createSearchClient()` or `createProductSyncWorker()`.

**Fix:** Add initialization to `apps/api/src/index.ts` startup sequence:

```
server start
  → createSearchClient(MEILISEARCH_HOST, MEILISEARCH_API_KEY)
  → initProductIndex()  // creates "products" index if not exists
  → createProductSyncQueue(REDIS_URL)
  → createProductSyncWorker(REDIS_URL)
```

Wrap in try-catch so the API still starts if Meilisearch is unavailable (degraded mode — search returns empty results, product CRUD still works via Postgres).

### 1.2 Arabic Language Configuration (TASK-074)

Use Meilisearch v1.11 built-in Arabic support:

- **Localized attributes:** Configure `localizedAttributes` on the "products" index:
  - `titleAr`, `descriptionAr` → locale `ara`
  - `titleEn`, `descriptionEn` → locale `eng`
- **Stop words:** Configure Arabic stop words list (common prepositions, conjunctions: من, في, على, إلى, etc.)
- **Searchable attributes priority:** `[titleAr, titleEn, brand, descriptionAr, descriptionEn, sku]`
- **Filterable attributes:** `[categoryId, brand, status, basePrice, inStock]` (remove `averageRating` from both filterable list and `ProductSearchDocument` type — not implemented yet)
- **Sortable attributes:** `[basePrice, createdAt]` (remove `averageRating` from sortable list and `ProductSearchDocument` type)

No custom Arabic NLP pipeline in v1 — rely on Meilisearch built-in tokenization. Can iterate in a future sprint.

### 1.3 Typo Tolerance & Synonyms (TASK-075)

**Typo tolerance:** Meilisearch default is already good — 1 typo for 5-8 char words, 2 for 9+. Configure via `index.updateTypoTolerance()`:
- `enabled: true`
- `minWordSizeForTypos: { oneTypo: 5, twoTypos: 9 }`

**Synonym management:**

- `GET /api/v1/admin/search/synonyms` — list all synonym groups

### 1.4 Search API (TASK-076)

New route file `routes/v1/search.ts`:

**`GET /api/v1/search`**

Query params:
| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `q` | string | required | Search query (min 1 char) |
| `category_id` | string | — | Filter by category UUID |
| `brand` | string[] | — | Filter by brand names (OR within) |
| `price_min` | number | — | Minimum price |
| `price_max` | number | — | Maximum price |
| `in_stock` | boolean | — | Only in-stock products |
| `sort` | enum | `relevance` | `relevance`, `price_asc`, `price_desc`, `newest` |
| `limit` | number | 20 | Results per page (max 50) |
| `offset` | number | 0 | Offset-based pagination (Meilisearch uses offset, not cursor) |

Implementation in new service `services/search-api.ts`:
1. Build Meilisearch filter string from query params (AND across facets, OR within `brand[]`)
2. Build sort array from `sort` param
3. Call `index.search(q, { filter, sort, limit, offset, facets: ["brand", "categoryId", "inStock"], attributesToHighlight: ["titleEn", "titleAr"], facetStats: ["basePrice"] })`
4. Map Meilisearch hits to `ProductListItem` type (fetch full product data from Postgres for images/variants not in search index)
5. Return response envelope

Response shape:
```typescript
{
  success: true,
  data: ProductListItem[],
  meta: {
    total: number,
    query: string,
    offset: number,
    limit: number,
    facets: {
      brands: Array<{ value: string; count: number }>,
      categories: Array<{ id: string; nameEn: string; nameAr: string; count: number }>,
      priceRange: { min: number; max: number }  // from Meilisearch facetStats
    }
  }
}
```

**Redis caching:** Cache full search responses with key `search:{hash(q+filters+sort+offset+limit)}`, TTL 2 minutes. Invalidate on product index update (via BullMQ job completion hook).

### 1.5 Auto-suggest API (TASK-077)

**`GET /api/v1/search/suggest`**

Query params: `q` (string, min 2 chars)

Implementation:
1. Search products via Meilisearch: `index.search(q, { limit: 4, attributesToRetrieve: ["id", "titleEn", "titleAr", "slug"] })`
2. Search categories: Prisma `findMany` where `nameEn ILIKE %q%` OR `nameAr ILIKE %q%`, limit 2
3. Search brands: Prisma `distinct brand` from products where `brand ILIKE %q%`, limit 2
4. Combine into grouped response (max 8 total suggestions: 4 products + 2 categories + 2 brands)

Response shape:
```typescript
{
  success: true,
  data: {
    suggestions: Array<{
      type: "product" | "category" | "brand",
      textEn: string,
      textAr: string,
      url: string  // e.g., "/products/iphone-16" or "/category/electronics" or "/search?q=Samsung&brand=Samsung"
    }>
  }
}
```

Target: <200ms p95 response time.

## Phase 2: Cart Backend (TASK-078)

### 2.1 Cart Service

New service `services/cart.ts` with functions:

**`getOrCreateCart(userId?, sessionId?)`** — Find existing cart or create new one. Auth users: lookup by userId. Guests: lookup by sessionId.

**`getCartWithDetails(cartId)`** — Fetch cart with items joined to product variants, products, and first image. Returns fully hydrated cart for API response.

**`addItem(cartId, productVariantId, quantity)`**:
1. Validate variant exists, is active, product is published
2. Check stock: `variant.stockQuantity >= requestedQuantity` (if item already in cart, check `existingQty + requestedQty <= stock`)
3. If variant already in cart → increment quantity
4. Else → create new CartItem with `unitPriceAtAddition` snapshot
5. Call `recalculateCart(cartId)`

**`updateItemQuantity(cartId, itemId, quantity)`**:
1. If quantity === 0 → remove item
2. Validate stock for new quantity
3. Update quantity
4. Call `recalculateCart(cartId)`

**`removeItem(cartId, itemId)`**:
1. Delete CartItem
2. Call `recalculateCart(cartId)`

**`recalculateCart(cartId)`**:
```
items = fetch all cart items with current variant prices
subtotal = SUM(items.map(item => {
  effectivePrice = item.variant.priceOverride ?? item.variant.product.basePrice
  return effectivePrice * item.quantity
}))
taxAmount = subtotal * 0.15
shippingEstimate = subtotal >= 500 ? 0 : 30
discountAmount = 0  // placeholder
grandTotal = subtotal + taxAmount + shippingEstimate - discountAmount
UPDATE cart SET subtotal, taxAmount, shippingEstimate, discountAmount, grandTotal, lastActivityAt=now()
```

**Why `unitPriceAtAddition` exists but isn't used in totals:** `unitPriceAtAddition` is an audit/history field — it records the price when the user added the item (useful for analytics, price-change notifications, and future "price dropped" features). However, `recalculateCart()` always uses the **current** `variant.priceOverride ?? product.basePrice` so the cart total reflects live pricing. **Implementers must NOT use `unitPriceAtAddition` in total calculations.**

**`mergeCarts(userId, guestSessionId)`**:
1. Find guest cart by sessionId
2. Find/create user cart by userId
3. For each guest item: if variant exists in user cart → keep higher quantity; else → move item to user cart
4. Recalculate user cart
5. Delete guest cart

### 2.2 Cart API Routes

New route file `routes/v1/cart.ts`:

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| `GET` | `/api/v1/cart` | Optional | Get current cart (auth or session) |
| `POST` | `/api/v1/cart/items` | Optional | Add item to cart |
| `PATCH` | `/api/v1/cart/items/:itemId` | Optional | Update item quantity |
| `DELETE` | `/api/v1/cart/items/:itemId` | Optional | Remove item from cart |
| `POST` | `/api/v1/cart/merge` | Required | Merge guest cart into user cart |

**Session handling middleware:**
- If request has valid JWT → use `userId`
- Else → read `cart_session` cookie; if missing, generate UUID and set cookie (HTTP-only, 30-day expiry, SameSite=Lax)
- Pass `userId` and/or `sessionId` to cart service

**Cart response shape:**
```typescript
{
  success: true,
  data: {
    id: string,
    items: Array<{
      id: string,
      quantity: number,
      variant: {
        id: string,
        sku: string,
        attributes: Record<string, string>,
        stockQuantity: number,
        effectivePrice: number  // priceOverride ?? basePrice
      },
      product: {
        id: string,
        titleEn: string,
        titleAr: string,
        slug: string,
        image: { url: string, altTextEn: string } | null
      }
    }>,
    subtotal: number,
    taxAmount: number,
    shippingEstimate: number,
    discountAmount: number,
    grandTotal: number,
    itemCount: number  // total quantity across all items
  }
}
```

### 4.3 Cart State Management (TASK-085)

**This task also includes connecting the existing cart page (`/cart`) to real data** — replacing the placeholder items in `app/[locale]/cart/page.tsx` with data from `useCart()`.

**Files:**
- `lib/cart-context.tsx` — CartProvider with React Query integration
- `lib/api/cart.ts` — API client functions
- `lib/query-keys.ts` — add `cart` key factory
- `app/[locale]/cart/page.tsx` — connect to real cart data via `useCart()`

**CartProvider:**
```typescript
// Wraps app in layout.tsx, inside AuthProvider and ReactQueryProvider
<CartProvider>
  {children}
</CartProvider>
```

**Exposed via `useCart()` hook:**
- `cart` — full cart data (items, totals)
- `itemCount` — total quantity
- `isLoading` — initial fetch loading
- `addItem(productVariantId, quantity)` — mutation with optimistic itemCount update
- `updateQuantity(itemId, quantity)` — mutation with optimistic update
- `removeItem(itemId)` — mutation with optimistic removal
- `isAddingItem` — loading state for add mutation

**Optimistic updates:**
- `addItem`: increment `itemCount` immediately, invalidate cart query on success/error
- `updateQuantity`: update item quantity in cache immediately, rollback on error
- `removeItem`: remove item from cache immediately, rollback on error

**Guest-to-auth merge:**
- In `AuthProvider`, after successful login:
  1. Check if `cart_session` cookie exists (via API call or cookie read)
  2. If yes → call `POST /api/v1/cart/merge` with sessionId
  3. Invalidate cart query to refetch merged cart
  4. Server clears the session cookie in merge response

**API client (`lib/api/cart.ts`):**
- `fetchCart()` → `GET /api/v1/cart` via `apiClient` (browser, includes cookies)
- `addCartItem(data)` → `POST /api/v1/cart/items`
- `updateCartItem(itemId, data)` → `PATCH /api/v1/cart/items/${itemId}`
- `removeCartItem(itemId)` → `DELETE /api/v1/cart/items/${itemId}`
- `mergeCart(sessionId)` → `POST /api/v1/cart/merge`
