## Implementation Phases

### Phase A — Backend Auth APIs (8 tasks)

**Order and rationale:** Each task builds on the previous. Registration creates users → login authenticates them → lockout protects them → password reset recovers them → MFA secures admins → audit logs everything → profile lets them self-manage.

#### TASK-030: Email Registration API
- **Endpoints:** `POST /api/v1/auth/register/email`, `POST /api/v1/auth/verify-email`, `POST /api/v1/auth/resend-verification`
- **Validation:** Zod schema — email format + uniqueness, password (min 8, 1 upper, 1 number), first/last name, preferred language (ar/en). Validate reCAPTCHA token on registration.
- **Flow:** Validate reCAPTCHA → validate input → hash password (bcrypt 12) → create user (status: `pending_verification`) → generate token (crypto.randomUUID) → store in Redis (24h TTL) → log verification URL to console
- **Verification:** Validate token from Redis → set `email_verified_at` → update status to `active` → delete token
- **Resend:** `POST /api/v1/auth/resend-verification` — accepts email, generates new token if user is `pending_verification`, rate limited to 3/hour
- **Errors:** 409 duplicate email, 400 weak password, 410 expired token (with resend option)

#### TASK-031: Phone Registration API
- **Endpoints:** `POST /api/v1/auth/register/phone`, `POST /api/v1/auth/verify-otp`
- **OTP service:** Generate 6-digit code → store in Redis (`otp:{phone}`, 5-min TTL) → track resend count (`otp-resend:{phone}:{sessionId}`, 3 max) → 30-sec cooldown between resends
- **Validation:** +966 format (Saudi mobile), first name required. Validate reCAPTCHA token.
- **Flow:** Validate reCAPTCHA → validate phone uniqueness → generate OTP → store in Redis → log OTP to console → return session ID
- **Verification:** Validate OTP → create user (status: `active`, phone verified) → delete OTP keys

#### TASK-032: Login API
- **Endpoints:** `POST /api/v1/auth/login/email`, `POST /api/v1/auth/login/phone`
- **Email login:** Validate reCAPTCHA → find user by email → verify password → check status (active, not locked) → issue tokens
- **Phone login:** Validate reCAPTCHA → find user by phone → send OTP → verify OTP → issue tokens
- **Token issuance:** Access token (JWT, 15min, in response body) + Refresh token (opaque, 7d, HTTP-only secure cookie)
- **Login events:** Record to `LoginEvent` table (new model, see Schema Migration section): user_id, email/phone, success/failure, IP, user_agent, failure_reason, timestamp

#### TASK-033: Account Lockout
- **Redis keys:** `login-attempts:{userId}` (counter, 30-min TTL), `account-locked:{userId}` (boolean, 30-min TTL)
- **Logic:** On failed login → increment counter → if counter >= 5 → set lock key → log lockout notification to console (mock email in Phase 1) → return 423 with `locked_until` timestamp
- **On success:** Delete attempt counter
- **On password reset:** Delete both keys (lift lockout)
- **Notification:** Log lockout event to console (real email delivery deferred to Phase 2 notification service)

#### TASK-034: Password Reset
- **Endpoints:** `POST /api/v1/auth/forgot-password`, `POST /api/v1/auth/reset-password`
- **Forgot:** Always return 200 (don't reveal email existence) → if email exists, generate token (1h TTL in Redis), log reset link
- **Reset:** Validate token → verify password complexity → hash → update user → delete all refresh tokens (invalidate sessions) → lift lockout → delete token
- **Rate limit:** `password-reset:{email}` counter in Redis, max 3/hour

#### TASK-035: Admin MFA Login
- **Endpoints:** `POST /api/v1/admin/auth/login`, `POST /api/v1/admin/auth/mfa/setup`, `POST /api/v1/admin/auth/mfa/verify`
- **Login step 1:** Validate email + password → verify user has admin role → return `mfa_required: true` with temp session token (5-min TTL)
- **MFA setup:** Generate TOTP secret (otpauth library) → store encrypted → return QR code URI (qrcode library) → require one valid code to confirm setup
- **MFA verify:** Validate TOTP code against secret → issue JWT with admin claims → log admin login event
- **Lockout:** 3 failed MFA → lock admin 30 min (same pattern as user lockout)
- **IP allow-list:** Skeleton middleware that checks request IP against configurable allow-list (env var or DB). Disabled by default. When enabled, non-listed IPs receive 403 with security alert logged.

#### TASK-036: Audit Log Service
- **Table:** Already exists in schema (`AuditLog`)
- **Middleware:** Express middleware for admin routes that captures before/after state
- **Service methods:** `recordAction(actorId, actorIp, action, entityType, entityId, before, after)`, `queryLogs(filters, pagination)`
- **Endpoint:** `GET /api/v1/admin/audit-logs` with filters (actor, action, entity, date range) + cursor pagination

#### TASK-037: User Profile API
- **Endpoints:** `GET /api/v1/users/me`, `PATCH /api/v1/users/me`, `POST /api/v1/users/me/change-email`, `POST /api/v1/users/me/change-password`, `POST /api/v1/users/me/change-phone`
- **PATCH:** Update first_name, last_name, preferred_language only (date_of_birth and gender deferred — fields exist on model but not in Sprint 1.1 scope per dev plan)
- **Change email:** Send verification to new email → on verify, update email, notify old email
- **Change password:** Require current password → hash new → invalidate other sessions
- **Change phone:** Send OTP to new phone → on verify, update phone

### Phase B — Frontend Auth Pages (6 tasks)

**Order:** Layout shell first (TASK-040) since all pages nest inside it. Then registration (038), login (039), password reset (041). Admin layout (043) then admin login (042).

#### TASK-040: Storefront Layout Shell
- Header with logo, search placeholder, language toggle, user dropdown, cart badge
- Footer with links, social icons, payment method icons
- Mobile: hamburger menu + sticky bottom nav (Home, Categories, Cart, Account)
- Full RTL mirroring with CSS logical properties (`margin-inline-start` not `margin-left`)
- Responsive: 320px (mobile), 768px (tablet), 1024px (desktop)

#### TASK-038: Registration Page
- Two-tab form (Email / Phone) at `/[locale]/register`
- Email tab: email, password (with strength indicator), first name, last name
- Phone tab: +966 input with mask, first name
- reCAPTCHA v3 (invisible), social login buttons (Google, Apple)
- Client-side Zod validation mirroring server schemas
- React Query mutations, loading states, error display
- On success: email tab shows "check your email", phone tab shows OTP input

#### TASK-039: Login Page
- Email + password form at `/[locale]/login`
- Phone + OTP option (tab or toggle)
- Social login buttons (Google, Apple)
- "Forgot Password" link, "Create Account" link
- Locked account: show remaining time
- reCAPTCHA v3 (with v2 fallback on low score)

#### TASK-041: Password Reset UI
- `/[locale]/forgot-password` — email input, always shows success message
- `/[locale]/reset-password?token=...` — new password + confirm, strength indicator

#### TASK-043: Admin Layout Shell
- Left sidebar: collapsible, nav items with icons
- Top header: admin name, role badge, notification bell placeholder, language toggle, logout
- Breadcrumbs in main content area
- Permission-based nav item visibility (read from JWT claims)

#### TASK-042: Admin Login + MFA
- Step 1: email + password form
- Step 2: 6-digit code input (auto-focus each digit, paste support, auto-submit on 6th digit)
- First-time MFA setup: QR code display, verify code to confirm
- Error states: invalid credentials, locked account, invalid MFA

### Phase C — Integration (2 tasks)

#### TASK-044: Auth Context + JWT Flow
- `AuthProvider` (React Context): stores access token in memory (NOT localStorage), user object, loading state
- `useAuth()` hook: `user`, `isAuthenticated`, `login()`, `logout()`, `isLoading`
- API client interceptor: attach `Authorization: Bearer` header, intercept 401 → call `/auth/refresh` → retry original request → if refresh fails → logout
- React Query: `enabled: !!user` for auth-dependent queries

#### TASK-045: Social Login E2E
- Wire Google/Apple buttons to `GET /api/v1/auth/oauth/{provider}/initiate`
- Handle redirect callback: extract auth code → exchange for JWT → update auth context
- Handle errors: user cancellation, provider error, email conflict (prompt to link accounts)

### Phase D — Testing (3 tasks)

#### TASK-046: Backend Unit Tests
- Test auth service: password hashing, JWT generation/validation, refresh rotation, reuse detection
- Test OTP service: generation, validation, expiry, rate limiting
- Test lockout service: counter increment, lock trigger, reset, password reset lift
- Test MFA service: TOTP generation, verification, lockout
- Test audit service: log creation, JSONB diff
- Mock Redis and Prisma. Target: >90% coverage on auth modules.

#### TASK-047: API Integration Tests
- Supertest against real test database (separate test DB or transaction rollback)
- Test complete flows: register → verify → login → refresh → logout
- Test error paths: duplicate email, weak password, wrong OTP, locked account
- Test bilingual errors (Accept-Language header)

#### TASK-048: Frontend Component Tests
- React Testing Library + Vitest
- Test forms: validation, submission, error display, loading states
- Test in both AR (RTL) and EN (LTR) locales
- Test accessibility: labels, ARIA attributes, keyboard navigation

### Phase E — DevOps (2 tasks)

TASK-049 and TASK-050 (staging deployment and CI verification) will be addressed after all feature code is complete, as they depend on a working codebase.

## Technical Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Token storage (frontend) | In-memory (React state) | More secure than localStorage; refresh token in HTTP-only cookie handles persistence |
| OTP delivery | Console log (Phase 1) | Real SMS deferred to Phase 2; mock enables full flow testing |
| TOTP library | `otpauth` | Lightweight, well-maintained, supports RFC 6238 |
| QR generation | `qrcode` | Standard library, works server-side |
| Password hashing | bcrypt, 12 rounds | Already implemented in Phase 0 auth service |
| Test framework | Vitest (not Jest — dev plan references Jest but project uses Vitest) | Already configured in Phase 0, 48 passing tests |
| Test DB strategy | Transaction rollback per test | Fast, isolated, no cleanup needed |
| API versioning | `/api/v1/` prefix | Already established in Phase 0 |
| Commit convention | `feat(api): TASK-030 email registration` | Conventional commits with task reference |
