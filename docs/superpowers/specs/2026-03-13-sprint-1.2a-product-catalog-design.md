## 1. Schema Changes

### New field on Product model

```prisma
specifications Json? @default("{}")
```

Structure: `{"Material": {"en": "Cotton", "ar": "قطن"}, "Weight": {"en": "200g", "ar": "200 غرام"}}`

Bilingual values at the attribute level. Rendered as a key-value table on PDP.

### New SlugRedirect model

```prisma
model SlugRedirect {
  id         String   @id @default(uuid()) @db.Uuid
  entityType String   @map("entity_type")  // "category" | "product"
  oldSlug    String   @unique @map("old_slug")
  newSlug    String   @map("new_slug")
  createdAt  DateTime @default(now()) @map("created_at")

  @@map("slug_redirects")
}
```

Middleware checks this table on 404 for product/category routes — if old slug matches, returns 301 to the new URL. The `@unique` constraint on `oldSlug` already creates an index, so no separate `@@index` is needed.

### Fix: ProductVariant.product relation

The current schema has `product Product[]` (implicit many-to-many) on ProductVariant, which is a bug. The migration must fix this to a standard many-to-one:

```prisma
product Product @relation(fields: [productId], references: [id], onDelete: Cascade)
```

### Add `updatedAt` to ProductImage model

The existing `ProductImage` model lacks `updatedAt`. Since images can have their `sort_order` and alt text updated, add:

```prisma
updatedAt DateTime @updatedAt @map("updated_at")
```

### Image variant URL convention

Rather than adding `thumbnailUrl`/`mediumUrl`/`largeUrl` columns to `ProductImage`, we use a **path convention**. The single `url` field stores the base path (e.g., `/uploads/products/{productId}/{filename}`), and variants are derived by suffix:
- Original: `/uploads/products/{id}/image.jpg`
- Thumbnail (200px): `/uploads/products/{id}/image-thumb.webp`
- Medium (600px): `/uploads/products/{id}/image-medium.webp`
- Large (1200px): `/uploads/products/{id}/image-large.webp`

The service generates all variants on upload. The frontend constructs variant URLs from the base URL. This avoids extra schema columns and translates cleanly to S3 key prefixes later.

### Migration

Single Prisma migration:
1. Add `specifications` Json column to `products` table
2. Create `slug_redirects` table
3. Fix `product_variants` → `products` relation from many-to-many to many-to-one
4. Add `updated_at` column to `product_images` table

No data migration needed (greenfield).

## 2. Backend API Architecture

### Route Structure

```
routes/v1/
  admin/
    categories.ts    → /api/v1/admin/categories
    products.ts      → /api/v1/admin/products
    inventory.ts     → /api/v1/admin/inventory
    customers.ts     → /api/v1/admin/customers
  categories.ts      → /api/v1/categories       (public)
  products.ts        → /api/v1/products          (public)
```

Admin routes use `authenticate` + `requirePermission('resource:action')` middleware (from `middleware/auth.ts`). Public routes are unauthenticated with general rate limiting.

### Required Permissions (seed in migration)

| Permission | Roles |
|------------|-------|
| `categories:read` | Super Admin, Product Manager |
| `categories:create` | Super Admin, Product Manager |
| `categories:update` | Super Admin, Product Manager |
| `categories:delete` | Super Admin |
| `products:read` | Super Admin, Product Manager |
| `products:create` | Super Admin, Product Manager |
| `products:update` | Super Admin, Product Manager |
| `products:delete` | Super Admin |
| `inventory:read` | Super Admin, Product Manager, Order Manager |
| `inventory:update` | Super Admin, Product Manager |
| `customers:read` | Super Admin, Support Agent, Order Manager |

These are seeded as part of the Prisma migration or seed script update.

### Service Layer

```
services/
  category.ts      — CRUD, tree building, materialized path, slug redirect
  product.ts       — CRUD, status transitions, validation, soft-delete
  variant.ts       — CRUD, atomic stock operations, SKU validation
  product-image.ts — upload to local storage, resize with Sharp, CRUD
  inventory.ts     — stock updates with reason codes, low-stock detection
  customer.ts      — read-only queries with computed metrics
```

### Key Patterns

- **Cursor-based pagination:** Uses Prisma's native `id`-based cursor (`cursor` + `skip: 1`) consistent with existing `queryAuditLogs` pattern. For custom sort orders (e.g., price), encode compound cursor as opaque base64 string fitting the shared `PaginationMeta.cursor` type
- **Zod schemas:** Request validation in `schemas/category.ts`, `schemas/product.ts`, etc.
- **Audit logging:** All admin mutations logged via existing `recordAuditLog()` service
- **Redis caching:** Public category tree (5-min TTL) and product detail (5-min TTL), invalidated on mutations
- **BullMQ jobs:** Meilisearch sync dispatched on product create/update/delete/publish, processed by worker that indexes product + variants + category data

### Image Handling (Local Storage)

- Upload via `multer` middleware (multipart/form-data)
- Sharp resizes to 3 variants: thumbnail (200px), medium (600px), large (1200px) + WebP
- Stored in `uploads/products/{productId}/` directory with path convention (see Section 1)
- Served via Express static middleware at `/uploads/`
- Service interface: `uploadImage(file) → { url }` (variants derived by path convention)
- Max 10 images per product, max 10MB per file
- Accepted formats: JPEG, PNG, WebP
- Service abstracted behind `ImageStorageProvider` interface for easy S3 swap later

### Atomic Stock Operations

```sql
UPDATE product_variants
SET stock_quantity = stock_quantity - ?
WHERE id = ? AND stock_quantity >= ?
```

Prisma raw query for atomic decrement. Returns affected rows — 0 means insufficient stock.

## 3. Admin Frontend Architecture

### New Dependencies

| Package | Purpose |
|---------|---------|
| `@tiptap/react`, `@tiptap/starter-kit`, `@tiptap/extension-text-align` | Rich text editor for product descriptions |
| `react-hook-form` + `@hookform/resolvers` | Form management with Zod validation |
| `@tanstack/react-query` | Data fetching (matches storefront pattern) |
| `@dnd-kit/core` + `@dnd-kit/sortable` | Drag-and-drop for image reordering and category tree |

### Page Structure

```
apps/admin/src/app/dashboard/
  products/
    page.tsx              — product list with filters, search, quick actions
    new/page.tsx          — product creation form
    [id]/page.tsx         — product edit form (same component, pre-filled)
  categories/
    page.tsx              — category tree management
  inventory/
    page.tsx              — inventory dashboard with inline editing
  customers/
    page.tsx              — customer list (read-only)
    [id]/page.tsx         — customer detail
```

### Product Form (TASK-058)

- **Tabbed layout:** Basic Info | Description | Pricing | Variants | Images | SEO
- **AR/EN fields side-by-side** (two columns) for easy comparison
- **Auto-save draft** every 30 seconds (debounced React Query mutation)
- **Slug auto-generates** from English title, editable
- **Category selector:** hierarchical dropdown fetched from API
- **Status bar** at top showing Draft/Published/Archived with transition buttons
- **Validation:** React Hook Form with Zod — required fields enforced before publish

### Variant Management (TASK-059)

- Attribute definition (add "Color", "Size" as variant axes)
- Variant matrix generator (auto-create all combinations)
- Per-variant fields: SKU, price override, stock, safety stock, low-stock threshold
- Variant-specific image assignment
- Inline editing for price and stock in variant grid
- Color coding: out-of-stock (red), low-stock (amber)

### Image Upload (TASK-060)

- Drag-and-drop upload (single and multiple) via `@dnd-kit`
- Upload progress indicator
- Image preview with thumbnails
- Reorder via drag-and-drop (updates sort_order)
- Alt text input (AR/EN) per image
- Variant association dropdown per image
- File validation: JPEG/PNG/WebP, max 10MB, max 10 images

### Product List (TASK-061)

- Data table: thumbnail, title, SKU, category, price, stock, status, date
- Filters: status dropdown, category dropdown, price range
- Search by title or SKU
- Sort by title, price, stock, date
- Cursor-based pagination
- Quick actions: publish, archive, delete (with confirmation)
- Optimistic updates via React Query

### Category Management (TASK-062)

- Tree view with expand/collapse (recursive React component)
- Drag-and-drop to reorder and reparent (max 4 levels enforced)
- Inline edit for category name
- Create/edit modal: name AR/EN, slug, parent, SEO metadata, banner image, active toggle, sort_order
- Product count badges per category node
- Delete flow: modal warns about children/products, requires reassignment target

### Inventory Dashboard (TASK-063)

- Summary cards: Total SKUs, In Stock, Low Stock, Out of Stock
- Data table with inline stock editing (click cell → input + reason dropdown → save)
- Tabs: All | Low Stock | Out of Stock
- Color-coded status badges (green/amber/red)
- Search by SKU or product name, filter by category

### Shared Patterns

- All list pages use React Query with URL-synced filters (query params)
- Optimistic updates for quick actions
- Loading skeletons during fetches
- Empty states with helpful messages
- Toast notifications for success/error (simple custom hook, no extra lib)

## 4. QA & Testing

### Unit Tests (TASK-069)

Location: `apps/api/src/__tests__/unit/`
Mock: Prisma client via vitest mocks
Target: >80% coverage

**Test coverage:**
- **Category service:** tree building, materialized path calculation, depth validation (max 4), slug redirect creation
- **Product service:** required field validation for publish, slug generation (including Arabic), status transitions (draft→published→archived), soft-delete logic
- **Variant service:** SKU uniqueness, price resolution (override vs base), stock status computation
- **Inventory service:** stock update with reason codes, low-stock threshold detection
- **Image service:** file validation (type, size), resize dimensions

### Integration Tests (TASK-070)

Location: `apps/api/src/__tests__/integration/`
Tool: Supertest against running Express app with test database

**Test scenarios:**
- Category CRUD lifecycle: create hierarchy (4 levels), update slug (verify redirect), delete with children (verify rejection), reassign and delete
- Product CRUD lifecycle: create draft → add images → add variants → publish (verify validation) → archive → soft-delete
- Product listing: filters (category, price, status), sorting, cursor pagination
- Inventory: update stock with reason code, verify InventoryMovement created, low-stock endpoint
- Customer list: filters, search, computed metrics
- Permission enforcement: buyer token rejected on admin endpoints, restricted admin roles blocked

### No Frontend Tests in 1.2a

Frontend component tests (TASK-071) and visual regression tests (TASK-072) are deferred to Sprint 1.2b alongside the storefront pages.
